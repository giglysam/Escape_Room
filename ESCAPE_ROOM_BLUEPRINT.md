# AI Escape Room Engine — Master Blueprint

> A complete, implementation-ready specification for building a Room‑Escape‑Maker–class
> escape room game where **every asset (background, props, character) is generated by AI**,
> **objects are placed coherently inside backgrounds the AI just produced**, and **each
> room is a chain of 6–10 small puzzles** instead of the classic single "find code → open
> door" loop.
>
> This document does NOT include code. It is the *contract* the engine, plan generator,
> asset pipeline, and UI must follow. If every section here is satisfied, the resulting
> game will look and feel like the REM (Room Escape Maker) and CSScape Room references
> the player provided.

## 1. Design pillars (non-negotiable)

These are the reasons earlier attempts felt "off". They are the acceptance criteria.

1. **No duplicate world objects.** The background is a *room with no interactive objects*
   (no painted door, no painted switch, no painted safe). Every interactive thing is a
   separate transparent cutout placed by the engine. ⇒ never see "two doors".
2. **Door is grounded.** Door's bottom edge sits exactly on the floor band. Never floating.
3. **Cup-on-table, not cup-on-floor.** Each prop has a *surface affinity* (`floor`,
   `table_top`, `wall_mid`, `wall_top`, `shelf`, `pedestal`, `ceiling`). Placement is
   constrained to anchor points the background was generated to contain.
4. **Multi-step chain per room (6–10 steps).** A room is solved by walking a *graph* of
   gated puzzles. Each step `gives` a flag; the next step `requires` it.
5. **Walking character with 4 directions.** Click a prop → character walks to that prop's
   interaction point → the interaction modal opens. Front / back / left / right sprites
   with a 2-frame walk cycle.
6. **Theme-faithful.** "Cyberpunk lab" must look cyberpunk in the background, in every
   prop, in modal text, in fonts/colors. Not a generic cabin with a cyberpunk cup on it.
7. **Responsive, REM-style chrome.** Top HUD (timer/mute/quit), framed canvas (16:9 1280×720),
   bottom nav bar (Turn L / Go L / Room N / Go R / Turn R), right-side compact inventory.
8. **Parallel image generation, never rate-limited.** Per-image rotating session/UA
   fingerprint, jittered concurrency cap, retry with backoff. Loading screen shows real
   progress (`12 / 47 generated`).

If a feature breaks any of these, it is a regression.

## 2. End-to-end pipeline

```
┌──────────────┐   theme   ┌─────────┐   plan.json   ┌────────────────┐   cutouts   ┌─────────────┐
│  Player UI   ├──────────▶│  LLM    ├──────────────▶│ Asset pipeline ├────────────▶│   Engine    │
│ "Cyberpunk"  │           │ (grok3) │  (deterministic│ (image-gen +    │             │ (canvas2D)  │
└──────────────┘           └─────────┘  fallback)     │  bg-removal +   │             └─────────────┘
                                                      │  trim + cache)  │
                                                      └────────────────┘
```

* **One LLM call** produces the *entire game plan*: rooms, narrative, objects, puzzles,
  flags, prompts. No live LLM during play.
* **N parallel image-gen calls** (cap = `GEN_CONCURRENCY = 5`) generate one PNG per object
  + one JPG per background. Each call rotates a fingerprint to dodge rate-limits.
* The engine never calls the network during play. It only consumes `plan.json` + cached
  assets.

This guarantees: zero blank screens, deterministic playback, easy retry.

## 3. The plan schema (single source of truth)

`plan.json` (produced once, cached in `localStorage`) is the contract between the LLM,
the asset pipeline, and the runtime engine.

```jsonc
{
  "version": 2,
  "theme": {
    "id": "cyberpunk-lab",
    "title": "Neon Vault 7",
    "hook":   "An anonymous SMS at 03:14 told you to enter the back door of the arcade.",
    "mission":"Reboot the rogue model before it broadcasts.",
    "stakes": "If the timer hits zero, the model leaks every face it has scanned.",
    "tags":   ["cyberpunk","neon","wet-asphalt","CRT","holograms"],
    "palette":{ "bg": "#06121f", "wall": "#0d2336", "floor":"#1a0d2a",
                "neon":"#ff2bd6", "rim":"#21f0ff" },
    "fonts":  { "title":"Orbitron", "body":"Rajdhani" }
  },
  "rooms": [
    {
      "id": "r1",
      "name": "Server Pit",
      "mission": "Get the master keycard.",
      "background_prompt": "<see §6.1>",
      "background_asset":  "bg/r1.jpg",
      "anchors": [
        { "id":"door_anchor",  "kind":"door_socket",  "x": 980, "y": 110, "w":200, "h":410 },
        { "id":"table_top_1",  "kind":"table_top",    "x": 240, "y": 380, "w":260, "h": 18 },
        { "id":"wall_mid_left","kind":"wall_mid",     "x":  60, "y": 220, "w":160, "h":160 },
        { "id":"shelf_top",    "kind":"shelf",        "x": 540, "y": 280, "w":260, "h": 14 },
        { "id":"pedestal_c",   "kind":"pedestal",     "x": 600, "y": 470, "w": 80, "h": 80 },
        { "id":"floor_back_l", "kind":"floor_back",   "x": 120, "y": 470, "w":120, "h":120 },
        { "id":"floor_front_r","kind":"floor_front",  "x": 900, "y": 560, "w":160, "h":140 }
      ],
      "objects": [
        {
          "id":"door",     "kind":"door",       "anchor":"door_anchor",
          "asset":"obj/r1_door.png",
          "prompt":"<see §6.2>",
          "states":["locked","unlocked","open"], "initial":"locked",
          "requires":["has_master_card"],
          "gives_on_open":["exit_r1"]
        },
        {
          "id":"safe",     "kind":"container",  "anchor":"pedestal_c",
          "asset":"obj/r1_safe.png",
          "puzzle":{ "type":"sequence_lock", "answer":"NEON" },
          "gives_on_solve":["safe_open"]
        },
        {
          "id":"keycard",  "kind":"item",       "anchor":"shelf_top",
          "asset":"obj/r1_keycard.png",
          "hidden_until":["safe_open"],
          "collect_to_inventory":"master_card",
          "gives_on_collect":["has_master_card"]
        },
        // ... 4–8 more for a 6–10 step chain ...
      ],
      "chain": [
        { "step":1, "trigger":"switch_breaker",   "gives":"power_on" },
        { "step":2, "trigger":"open_drawer",      "requires":"power_on", "gives":"has_screwdriver" },
        { "step":3, "trigger":"unscrew_vent",     "requires":"has_screwdriver", "gives":"vent_open" },
        { "step":4, "trigger":"collect_battery",  "requires":"vent_open", "gives":"has_battery" },
        { "step":5, "trigger":"power_keypad",     "requires":"has_battery", "gives":"keypad_live" },
        { "step":6, "trigger":"solve_keypad",     "requires":"keypad_live", "gives":"safe_open" },
        { "step":7, "trigger":"collect_keycard",  "requires":"safe_open",   "gives":"has_master_card" },
        { "step":8, "trigger":"unlock_door",      "requires":"has_master_card", "gives":"exit_r1" }
      ]
    }
    // r2, r3, r4 ...
  ],
  "endings": {
    "win":  "You jacked the antenna. The model goes dark. Sirens fade.",
    "lose": "The broadcast hits. Every screen in the city now wears your face."
  }
}
```

Key invariants:

* `anchors` are **placeholders the LLM committed to inside the background prompt**
  (see §6.1). They guarantee surface affinity.
* Every `object.anchor` must reference a real anchor of a compatible `kind`.
* Every `chain.requires` must be `gives` of an earlier step (DAG, no cycles).
* The last step of every room emits `exit_<roomId>` which the engine uses to advance.

## 4. Anchor system (the answer to "cup on table, not floor")

The background is generated to contain a fixed list of *empty surfaces*. The plan
generator chooses which surfaces it needs and tells both the LLM (so it draws them)
and the engine (so it places cutouts on them).

| anchor `kind`  | typical use        | y-band on 1280×720 | semantics                       |
| -------------- | ------------------ | ------------------ | ------------------------------- |
| `wall_top`     | mural, vent, sign  | 60 – 220           | high on wall                    |
| `wall_mid`     | poster, switch     | 200 – 380          | eye-level                       |
| `shelf`        | book, jar, idol    | 260 – 360          | horizontal surface, props on top|
| `table_top`    | cup, lamp, keypad  | 360 – 430          | horizontal surface, props on top|
| `pedestal`     | safe, totem        | 440 – 560          | small footprint, props on top   |
| `floor_back`   | crate, plant       | 440 – 580          | grounded, behind char           |
| `floor_front`  | rug, debris        | 560 – 700          | grounded, in front of char      |
| `door_socket`  | door cutout        | bottom = `FLOOR_BOT` | door bottom = floor             |
| `ceiling`      | hanging lamp       | 0 – 100            | ceiling fixture                 |

Placement rule: an object with `kind=item` whose `anchor.kind=table_top` is rendered
with its bottom edge = `anchor.y` (so it *sits on* the table, not in front of it).
For `pedestal`, the prop is centered on the pedestal. For `floor_*`, the prop's bottom
edge = `FLOOR_BOT` regardless of anchor height (the anchor only specifies x-band).

`door` is special: `bottomY = FLOOR_BOT`, `topY = FLOOR_BOT - doorH`. **Door never floats.**

## 5. Theme-faithful generation (no generic backgrounds)

The LLM expands a one-line theme into a full `theme` block (§3). All prompts are then
*templated* with theme tokens so the AI never produces generic content.

### 5.1 Theme expansion prompt (LLM)

> Given the user's theme `"<input>"`, return a JSON `theme` object with: `id`, `title`,
> a 1-sentence `hook` (cinematic, 2nd person), a 1-sentence `mission`, a 1-sentence
> `stakes`, 6–12 `tags` (visual nouns + textures), a `palette` (5 hex colors:
> `bg/wall/floor/neon/rim`), `fonts` (a title font + a body font from Google Fonts
> appropriate to the theme), and per-room `vibe_packs`: `threshold` (entry), `heart`
> (mid-game), `way_out` (climax). Forbidden: clichés ("dimly lit room", "old dusty"),
> generic adjectives, modern logos. Required: at least 3 sensory tokens (smell, sound,
> texture) per room.

### 5.2 Theme injection rule

Every image prompt downstream is built as:

```
{prompt_skeleton} | theme: {theme.tags joined ", "} | palette: {palette.* hex} |
forbidden: text, logos, watermark, people, UI, signature
```

This guarantees a Victorian study and a Mars station never look the same.

## 6. Image prompt templates (the actual magic)

### 6.1 Background prompt (per room)

The background is **an empty atmospheric room** that *contains the right surfaces but
no interactive props*. It must satisfy:

```
A wide-angle interior of a {theme.title}, {room.vibe_pack.heart}, {theme.tags ...},
view from human eye level, vanishing point near horizontal center,
floor band visible across the full width, walls visible left+center+right,
SURFACES PRESENT: {anchors.kind ...}.
PURE EMPTY ROOM.
NO door, NO door frame, NO switch, NO keypad, NO safe, NO mural with words,
NO posters with text, NO computer screen content, NO people, NO creatures,
NO UI, NO HUD, NO text, NO logos, NO watermark, NO signature.
Lighting: {theme.palette.neon} accent + {theme.palette.rim} rim light.
Style: photoreal painterly, 16:9, sharp focus, depth of field on the back wall.
```

Why this works:
- Lists the **anchors** so the AI paints surfaces (a table, a shelf, a pedestal slab,
  a vent rectangle) but
- Forbids the **interactive things** that would otherwise duplicate our cutouts.

### 6.2 Object prompt (per cutout)

```
A single {object.kind} for a {theme.title} escape room: {object.descriptor},
materials: {theme.tags}, colors: {theme.palette.neon}/{theme.palette.rim},
centered, full object visible, isometric or 3/4 view as appropriate to a {object.kind},
PURE FLAT SOLID WHITE #ffffff BACKGROUND, no environment, no room, no building,
no architecture, no people, no creatures, no shadow plate, no text, no logos.
Style: matches the background painting style.
```

Why pure white background: enables the engine's chroma-key + flood-fill background
remover (tolerance ≈ 50) to produce a clean transparent PNG that drops into the scene.

### 6.3 Character prompt (4 directions × 2 frames)

Either generate procedurally on canvas (cheap, deterministic) OR generate 8 sprites:

```
A pixel-art / painterly character, {theme.tags character vibe}, full body, no background,
PURE FLAT SOLID WHITE #ffffff BACKGROUND, facing {front|back|left|right},
walk frame {1|2}, same pose ratios across all 8 frames, height {CHARACTER_H}px.
```

Procedural is recommended for v1 because it never fails generation and looks consistent
across themes (use the theme palette to color the silhouette).

## 7. Asset pipeline (Vercel serverless)

### 7.1 `/api/generate-image`

* Accepts `{prompt, generatorType?, sessionId?}`.
* Maintains a **fingerprint pool** of 5–10 `{User-Agent, sec-ch-ua, lang, platform}`
  tuples. The `sessionId` (random per call) deterministically picks one.
* Per-session warm-up GET to `/?type=...` before POST to `/api/generate`.
* Downloads the resulting image, converts to a base64 data URL, returns to client.
* Retries with exponential backoff (4s, 8s, 16s, 32s) on 429/5xx.

### 7.2 `/api/plan`

* Accepts `{theme}`. If `GROK_OK`, calls grok3 with the prompt in §5.1 + §3 schema.
  Validates the returned JSON against the schema. If invalid or LLM unavailable, falls
  back to a **procedural plan generator** (deterministic, theme-driven) so the user
  never sees an error.

### 7.3 Client-side asset manager

* `pMap(plan.allImagePrompts, GEN_CONCURRENCY=5, generate)` runs N image gens in parallel.
* Reports `{done, total}` to the loading screen.
* For each cutout: load → `removeBackground(tolerance=50)` → `trimTransparent()` →
  cache the trimmed bitmap by `assetId` in `localStorage` (data URL).
* Backgrounds are cached as raw JPEG.
* If a generation fails after retries, substitutes a themed placeholder rectangle so
  the game can still ship a playable room.

## 8. Multi-step puzzle chains (the REM feel)

A room is a **DAG of 6–10 steps**. Each step has:

```
{ step: N, trigger: <interactionId>, requires: [flag...], gives: [flag...] }
```

### 8.1 Built-in interaction kinds

| `object.kind`     | UI on click                                | Effect                      |
| ----------------- | ------------------------------------------ | --------------------------- |
| `switch`          | Toggle visual + click sound                | Sets `gives` flags          |
| `item`            | "Pick up <name>"                           | Adds to inventory, gives flag |
| `container`       | Modal: "Open" requires `requires` items    | Reveals child objects (`hidden_until`) |
| `pedestal`        | Drop item from inventory onto pedestal     | Validates, gives flag       |
| `sequence_lock`   | Modal with N buttons; correct order wins   | Gives flag                  |
| `keypad`          | Modal numeric pad, validates `answer`      | Gives flag                  |
| `mural` / `clue`  | Modal shows the artwork + 1 sentence       | Pure information            |
| `door`            | Tries to open; blocked unless `requires` met | Gives `exit_<room>`         |

### 8.2 Example chain (Cyberpunk Server Pit)

```
1. flip breaker switch                 → power_on
2. drawer (container, requires power)  → has_screwdriver
3. unscrew vent (requires screwdriver) → vent_open
4. collect battery from vent           → has_battery
5. install battery in keypad           → keypad_live
6. solve keypad (clue: serial on poster) → safe_open
7. open safe (container)               → reveals keycard
8. collect keycard                     → has_master_card
9. swipe keycard at door reader        → door_unlocked
10. open door                          → exit_r1
```

Hard rules:
* Every flag is **produced exactly once** (no duplicate `gives`).
* Every `requires` flag must be `gives` of a *strictly earlier* step (DAG check at
  plan-generation time).
* If the chain has < 6 steps, the plan generator must inflate (add a `mural` clue, a
  `switch_clue`, a hidden item) until ≥ 6. If > 10, split into a second room.

### 8.3 Variety enforcement

The plan generator must include at least 4 *distinct* interaction kinds per room
(no chain made entirely of switches). Recommended distribution per room:

* 1× `switch` (kickoff)
* 1–2× `container` (drawer, safe, vent)
* 2–3× `item` (collect / combine)
* 1× `pedestal` OR `sequence_lock` (dexterity / observation)
* 1× `keypad` OR `mural+clue` (logic)
* 1× `door` (terminal)

## 9. Walking character

* 4 facings × 2 walk frames = 8 sprites (`CHARACTER_W=64`, `CHARACTER_H=120`).
* On click of an interactable prop:
  1. Compute the prop's *interaction point* (anchor.x + anchor.w/2, FLOOR_BOT - 20).
  2. Set `charTargetX`. Compute facing: `target.x < char.x ? "left" : "right"`.
     (Use `up` when target is `wall_top` / `ceiling`, `down` when target is
     `floor_front`.)
  3. Engine ticks `updateCharacter(dt)` → moves `charX` toward `charTargetX` at
     ~220 px/s, advancing walk-frame every 120 ms.
  4. On arrival (|Δx| < 4), fire `pendingInteract` (open the modal / collect / toggle).
* Z-ordering: character drawn AFTER `floor_back` props but BEFORE `floor_front` props,
  so a crate in front partially occludes the character (depth illusion).
* Tinting: a soft ambient overlay using `theme.palette.bg` at 12% opacity makes the
  character feel "in the scene".

## 11. Game state machine

```
menu  ──theme────▶ generating(asset progress) ──ready────▶ briefing(hook/mission/stakes/timer)
                                                              │
                                                              ▼
            ◀──onLose──  playing  ──exit_lastRoom──▶ win
                            │
                            └── onTimeUp ──▶ lose (with "continue anyway" CTA)
```

`playing` substate:

```
playing
 ├── idle          (no character action)
 ├── walking       (charX → charTargetX)
 ├── interacting   (modal open)
 └── transitioning (between rooms; fade-to-color on exit_r{n})
```

## 12. Performance & robustness rules

* Backgrounds: 1280×720 JPEG ≤ 220 KB target. Pre-resize on the client if larger.
* Cutouts: trim transparent margin; cap longest side at 480 px to keep DOM/CPU light.
* `localStorage` budget: keep < 4 MB; if exceeded, evict oldest theme.
* Every async generator call must have a *hard 60s timeout* + retries; never block the
  loading screen forever.
* On reload, the game restores: current room, flags, inventory, switchStates, sequence
  progress, pedestal slots, consumed items, secondsLeft — so accidental refresh is safe.
* The engine never throws on missing assets; it draws a labeled placeholder rect tinted
  to the theme palette, so playthroughs are always completable.

## 13. Acceptance test (one paragraph per pillar)

Use these as the QA checklist before merging any change.

1. **No duplicate door**: open DevTools → search the rendered scene for any "door"-like
   shape inside the background JPEG. There must be none.
2. **Door grounded**: pixel-inspect the door cutout's bottom edge — it must equal
   `FLOOR_BOT` (= 580 on 1280×720).
3. **Cup on table, not floor**: spawn a `cup` item; assert `cup.bottom === tableTop.y`
   ± 2 px and `cup.x` is within `tableTop.x ... tableTop.x+tableTop.w`.
4. **Multi-step**: every room's chain has `length >= 6`. Skipping any step makes the
   final door reject the player with a themed message.
5. **Walking char**: click a wall_top prop → char walks (x changes), then a modal
   appears. Click a floor_front prop → char walks the other direction. Sprite swaps
   every 120 ms during walk; idle sprite faces the last move direction.
6. **Theme fidelity**: change the theme to "Underwater submarine" and reload. The HUD
   font, palette, ambient tint, background tags, and prop materials must all change.
7. **REM chrome**: viewport 1440×900 shows the canvas centered, the right-side inventory
   visible, the bottom nav bar present, the top HUD present.
8. **Parallel asset gen**: load 47-asset plan; wall-clock time ≤ 3× single-asset time
   (proves concurrency=5 works); zero 429s in network panel.

If any of these fail, revert the change.

## 15. Authoring a new theme in 30 seconds

1. User types: `"Bioluminescent reef temple"`.
2. `/api/plan` → LLM expands to full `theme` block (§5.1) + 3-room plan (§3).
3. Client kicks off parallel image-gen for the ~45 prompts the plan referenced.
4. As assets stream back, the loading screen ticks `12 / 45`.
5. When done, `briefing` screen shows the hook ("You wake on a tidal altar, gills
   already forming…"), mission, stakes, timer.
6. Click **Begin** → engine loads room 1 → character walks in from the left → first
   prop's hover tooltip appears. The chain is now in the player's hands.

Result: an immersive, multi-step, theme-perfect escape room generated end-to-end by
AI, looking and playing like the REM and CSScape Room references — but with backgrounds,
props, character, narrative, and puzzles entirely synthesized for whatever theme the
player imagined.
